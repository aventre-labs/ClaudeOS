1. **extension-template/AGENTS.md** -- AI agent guidance for extension development (TPL-04):

   Structure the document with these sections:

   a. **Kernel Principles** (inherited from ClaudeOS core):
      - Extensions run inside code-server in the ClaudeOS container
      - Never modify Claude Code -- it runs stock in tmux, managed by the supervisor
      - Never fork code-server -- configure via extensions, product.json, and settings only
      - All state persists on /data volume. Extensions should use /data/extensions/{extension-name}/ for their storage
      - Single-user system: no multi-tenant concerns

   b. **Supervisor API Contract**:
      - Base URL: http://localhost:3100/api/v1 (container-internal, no auth needed)
      - Endpoints:
        - GET /health -- { status, version, uptime }
        - POST /sessions -- Create session { name?, workdir?, model?, flags? }
        - GET /sessions -- List all sessions
        - GET /sessions/:id -- Get single session
        - POST /sessions/:id/stop -- Stop session
        - DELETE /sessions/:id -- Kill session (returns scrollback)
        - POST /sessions/:id/input -- Send input { text }
        - GET /sessions/:id/output -- Capture output { scrollback? }
        - POST /sessions/:id/archive -- Archive session
        - POST /sessions/:id/revive -- Revive archived session
        - POST /secrets -- Create secret { name, value, category?, tags? }
        - GET /secrets -- List secrets (no values)
        - GET /secrets/:name -- Get secret value
        - PUT /secrets/:name -- Update secret
        - DELETE /secrets/:name -- Delete secret
        - POST /extensions/install -- Install extension
        - GET /extensions -- List extensions
        - GET /settings -- Get supervisor settings
        - PUT /settings -- Update supervisor settings
      - WebSocket: ws://localhost:3100/api/v1/ws
        - Subscribe: { type: "subscribe", sessionId: "..." }
        - Status events: { type: "status", sessionId: "...", status: "..." }
        - Output events: { type: "output", sessionId: "...", data: "..." }

   c. **Extension Development Patterns**:
      - Use `fetch()` for supervisor API calls (available in code-server's Node.js runtime)
      - Use WebSocket for real-time session monitoring
      - Store extension-specific data in /data/extensions/{extension-name}/
      - Use the secrets API for credentials (never store secrets directly)
      - Build with esbuild, package with vsce --no-dependencies
      - Test with Vitest

   d. **MCP Server Pattern** (for extensions that expose tools to Claude Code):
      - Place MCP server code in mcp-server/ directory
      - MCP server runs as a separate process managed by the supervisor
      - Use @modelcontextprotocol/sdk for tool definitions
      - Document the MCP lifecycle: supervisor starts MCP server, Claude Code connects via stdio

   e. **GitHub Actions CI** (recommended):
      - Build -> Test -> Package VSIX -> Upload as release asset
      - Tag releases with semver (v0.1.0, v0.2.0, etc.)
      - VSIX asset name: {extension-name}-{version}.vsix
