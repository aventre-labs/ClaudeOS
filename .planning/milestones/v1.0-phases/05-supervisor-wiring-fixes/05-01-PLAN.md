  <name>Task 1: Wire BootService invocation from index.ts</name>
  <files>supervisor/src/index.ts, supervisor/test/boot-wiring.test.ts</files>
  <behavior>
    - Test 1: When isDryRun is false and isConfigured() returns true, BootService.installExtensions() and BootService.startCodeServer() are called after server.listen()
    - Test 2: When isDryRun is false and isConfigured() returns false, BootService.serveSetupPage(port) is called BEFORE server.listen(), then after setup completes, server is built and boot continues
    - Test 3: When isDryRun is true, BootService is NOT instantiated and no boot sequence runs
  </behavior>
  <action>
    **Test file** (`supervisor/test/boot-wiring.test.ts`):
    Since index.ts runs `main()` as a side-effect, testing it directly is awkward. Instead, extract the boot logic into a testable function. Refactor index.ts to export a `boot(options)` function that the test can call with mocked dependencies, while `main()` remains the thin CLI entry point that calls `boot()`.

    Alternatively (simpler, matches existing test patterns): Write an integration-style test that mocks BootService and buildServer, then calls a refactored `main` export. The test should:
    1. Mock `./services/boot.js` to provide a fake BootService with spy methods
    2. Mock `./server.js` to provide a fake buildServer returning a mock Fastify instance
    3. Verify that for non-dry-run: BootService is constructed, isConfigured() is checked, and installExtensions() + startCodeServer() are called
    4. Verify that for dry-run: BootService is not constructed

    **Production fix** (`supervisor/src/index.ts`):
    Use the two-phase boot approach from research (Option 2 -- cleanest):

    1. Import BootService from `./services/boot.js`
    2. Before `buildServer()`, if NOT isDryRun:
       a. Create a temporary ExtensionInstaller (needs dataDir only)
       b. Create BootService with `{ dataDir, extensionInstaller, setBootState: no-op, logger: console }`
       c. Check `bootService.isConfigured()` -- if false, call `await bootService.serveSetupPage(port)` (this blocks until user submits password, then closes setup server, freeing the port)
    3. After the isConfigured check (or if already configured), proceed with `buildServer()` and `server.listen()` as today
    4. After `server.listen()` succeeds, if NOT isDryRun:
       a. Create a NEW BootService using server decorators: `server.extensionInstaller`, `server.setBootState`, and `server.log` (bound with `.bind(server.log)` to avoid Pino `this` context issues -- see pitfall 2 in research)
       b. Call `await bootService.installExtensions()`
       c. Call `await bootService.startCodeServer()`

    This splits boot into pre-server (first-boot setup page) and post-server (extension install + code-server launch). No port conflict because serveSetupPage closes before server.listen binds.

    Note: ExtensionInstaller constructor only needs dataDir. Import it from `./services/extension-installer.js`.
  </action>

    - Test 2: On a server where auth.json exists (isDryRun=true uses test helper), GET /api/v1/secrets returns 200 (existing behavior preserved)
    - Test 3: POST /api/v1/secrets on a server without auth.json returns 503
  </behavior>
  <action>
    **Approach:** Always register secretRoutes. Change the route options interface to accept `dataDir` instead of `secretStore`. Routes lazily create SecretStore on first request where auth.json exists. If auth.json is missing, return 503.

    **Step 1 -- Update SecretStore** (`supervisor/src/services/secret-store.ts`):
    Add a static factory method that does not throw:
    ```typescript
    static tryCreate(dataDir: string): SecretStore | null {
      const authPath = join(dataDir, "config", "auth.json");
      if (!existsSync(authPath)) return null;
      try {
        return new SecretStore(dataDir);
      } catch {
        return null;
      }
    }
    ```
    Keep the existing constructor unchanged (still throws) for backward compatibility with code that expects a guaranteed store.

    **Step 2 -- Update SecretRouteOptions** (`supervisor/src/routes/secrets.ts`):
    Change the interface and route plugin to accept `dataDir` instead of a pre-built `secretStore`:
    ```typescript
    export interface SecretRouteOptions {
      dataDir: string;
    }
    ```
    Inside the plugin, lazily resolve the store:
    ```typescript
    let cachedStore: SecretStore | null = null;
    function getStore(): SecretStore | null {
      if (!cachedStore) {
        cachedStore = SecretStore.tryCreate(options.dataDir);
      }
      return cachedStore;
    }
    ```
    Each route handler calls `getStore()`. If null, reply 503 `{ error: "Secrets unavailable — complete first-boot setup", statusCode: 503 }`. Otherwise proceed with existing logic.

    **Step 3 -- Update server.ts** (`supervisor/src/server.ts`):
    Remove the conditional `if (secretStore)` block (lines 112-117). Replace with unconditional registration:
    ```typescript
    await server.register(secretRoutes, {
      prefix: "/api/v1",
      dataDir: options.dataDir,
    });
    ```
    Also remove the `secretStore` variable creation (lines 81-83) and the `server.decorate("secretStore", secretStore)` line (line 136). The secretStore is no longer managed at the server level -- it lives inside the routes.

    Keep the `existsSync(authPath)` check ONLY for the dry-run auth.json generation block (lines 55-61) which is unrelated.

    **Step 4 -- Test file** (`supervisor/test/routes/secrets-unconditional.test.ts`):
    Create a new test file that builds a server with `isDryRun: false` and a temp dataDir that has NO auth.json. This simulates a fresh container.

    - Verify GET /api/v1/secrets returns 503
    - Verify POST /api/v1/secrets returns 503
    - Then write an auth.json with a valid encryptionKey to the temp dir
    - Verify GET /api/v1/secrets now returns 200

    Note: Building with `isDryRun: false` means TmuxService (not DryRunTmuxService) will be created. The test only hits secrets routes so tmux is irrelevant, but if the constructor fails, mock TmuxService or use a try-catch setup. Check if TmuxService constructor throws without tmux binary -- if so, mock it.
