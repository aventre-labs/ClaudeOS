1. **Update flake.nix** -- Add supervisor build and container image derivations to the existing devShell-only flake from Plan 01:

   a. **Supervisor build derivation** using `pkgs.buildNpmPackage`:
      - pname: "claudeos-supervisor", version: read from supervisor/package.json
      - src: ./supervisor
      - npmDepsHash: Run `nix-prefetch` or leave as placeholder "sha256-AAAA..." with a comment to update after first build attempt
      - buildPhase: `npx esbuild src/index.ts --bundle --platform=node --format=cjs --outfile=dist/supervisor.cjs --external:@fastify/websocket`
      - installPhase: Copy dist/supervisor.cjs to $out/bin/, copy node_modules/@fastify/websocket to $out/lib/ (since it's externalized from the bundle)
      - NOTE: @fastify/websocket must be external because it contains native bindings (ws). Include its node_modules in the image.

   b. **Container image derivation** using `pkgs.dockerTools.buildLayeredImage`:
      - name: "ghcr.io/aventre-labs/claudeos"
      - tag: "latest" (or version from package.json)
      - contents: [ pkgs.nodejs_22, pkgs.tmux, pkgs.git, pkgs.code-server, pkgs.bash, pkgs.coreutils, pkgs.cacert, supervisor-derivation ]
      - For su-exec/gosu: Check if `pkgs.su-exec` exists. If not, use `pkgs.gosu` or fall back to `pkgs.util-linux` (provides setpriv). Per research open question #4.
      - fakeRootCommands (NOT runAsRoot -- works cross-platform):
        ```
        mkdir -p ./data/extensions ./data/sessions ./data/secrets ./data/config
        mkdir -p ./app
        cp ${./config/product.json} ./app/product.json
        cp ${./config/settings.json} ./app/settings.json
        cp ${./config/default-extensions.json} ./app/default-extensions.json
        cp ${./first-boot/setup.html} ./app/setup.html
        cp ${./entrypoint.sh} ./app/entrypoint.sh
        chmod +x ./app/entrypoint.sh
        ```
      - config.Entrypoint: [ "/app/entrypoint.sh" ]
      - config.ExposedPorts: { "8080/tcp" = {}; "3100/tcp" = {}; }
      - config.Env: [ "NODE_ENV=production", "CLAUDEOS_DATA_DIR=/data" ]
      - config.WorkingDir: "/app"

   c. **Claude Code installation**: Per research open question #1, Claude Code uses `curl -fsSL https://claude.ai/install.sh | bash`. This cannot run in a Nix build sandbox (no network). Two approaches:
      - Option A (preferred): Install Claude Code at container runtime in entrypoint.sh if not already present on /data volume. Cache the installation on persistent volume.
      - Option B: Create a fixed-output Nix derivation that downloads Claude Code. More Nix-pure but fragile if the install URL changes.
      - Go with Option A -- simpler, more robust. Document in entrypoint.sh.

   d. **Outputs structure**:
      ```nix
      packages.${system} = {
        default = supervisor;
        container = containerImage;
      };
      ```

2. **entrypoint.sh** -- Container entrypoint script:
   ```bash
   #!/bin/bash
   set -e

   # Ensure data directories exist and are owned by app user
   mkdir -p /data/extensions /data/sessions /data/secrets /data/config
   chown -R app:app /data

   # Install Claude Code if not already present
   if ! command -v claude &> /dev/null; then
     echo "[ClaudeOS] Installing Claude Code..."
     su-exec app bash -c 'curl -fsSL https://claude.ai/install.sh | bash'
     echo "[ClaudeOS] Claude Code installed"
   fi

   # Drop privileges and exec supervisor
   exec su-exec app node /path/to/supervisor.cjs
   ```
   - Use su-exec if available, otherwise setpriv from util-linux: `exec setpriv --reuid=app --regid=app --init-groups node /path/to/supervisor.cjs`
   - Script must be executable (chmod +x in fakeRootCommands)
   - The `app` user must be created in the Nix image. Use `fakeRootCommands` with `echo "app:x:1000:1000::/home/app:/bin/bash" >> /etc/passwd` and `echo "app:x:1000:" >> /etc/group` and `mkdir -p /home/app && chown 1000:1000 /home/app`
   - Handle the case where tmux needs a proper TERM variable: set TERM=xterm-256color in config.Env

**Note on Nix build validation:** Building the container image requires `nix build .#container --system x86_64-linux`. On macOS, this requires a remote Linux builder or NixOS VM. If building locally is not possible, validate that flake.nix is syntactically correct with `nix flake check` and document the build requirement.

1. **docker-compose.yml** -- Local development:
   ```yaml
   version: "3.8"
   services:
     claudeos:
       image: ghcr.io/aventre-labs/claudeos:latest
       ports:
         - "8080:8080"   # code-server UI
         - "3100:3100"   # supervisor API
       volumes:
         - claudeos-data:/data
       environment:
         - NODE_ENV=development
         - CLAUDEOS_DATA_DIR=/data
         - TERM=xterm-256color
       restart: unless-stopped
       healthcheck:
         test: ["CMD", "curl", "-f", "http://localhost:3100/api/v1/health"]
         interval: 30s
         timeout: 10s
         retries: 3
         start_period: 60s

   volumes:
     claudeos-data:
   ```
   - Image is loaded via `nix build .#container && docker load < result` before compose up
   - Add a comment at the top explaining the Nix build + docker load workflow
   - start_period is 60s to allow first-boot setup time

2. **railway.toml** -- Railway deployment:
   ```toml
   [build]
   # Railway pulls pre-built image from GHCR (no build step)
   # Push image: nix build .#container && docker load < result && docker push ghcr.io/aventre-labs/claudeos:latest
   dockerImage = "ghcr.io/aventre-labs/claudeos:latest"

   [deploy]
   healthcheckPath = "/api/v1/health"
   healthcheckTimeout = 120
   restartPolicyType = "ON_FAILURE"
   restartPolicyMaxRetries = 3

   [[deploy.volumes]]
   mount = "/data"
   ```
   - Railway-specific: healthcheckPath uses the supervisor API port (Railway routes to the correct internal port based on service config)
   - healthcheckTimeout is 120s to allow first-boot setup
   - Volume mounted at /data for persistence
   - Note: Railway needs PORT env var set to 8080 for the public-facing service (code-server)
