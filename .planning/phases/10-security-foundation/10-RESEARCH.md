# Phase 10: Security Foundation - Research

**Researched:** 2026-03-15
**Domain:** Race condition prevention, Railway auth integration, deploy button, GitHub Actions fork workflow
**Confidence:** HIGH

## Summary

Phase 10 addresses two requirements: SETUP-04 (race condition protection for first-boot setup) and DEPLOY-01 (deploy button works for any fork). The current `BootService` has no race condition protection -- two simultaneous POST requests to `/api/v1/setup` can both succeed, creating a TOCTOU vulnerability. The fix is straightforward: use an atomic lock mechanism (either an in-memory mutex or a filesystem lock file) to ensure only one setup request proceeds.

The deploy button requires switching from a hardcoded Railway template ID to the `?template=` URL parameter format that accepts a GitHub repo URL. A GitHub Action will auto-patch the README in forked repos. The auth model changes are significant: `auth.json` is eliminated, `CLAUDEOS_AUTH_TOKEN` becomes the source of truth (auto-generated by Railway via `${{secret(32)}}`), and the encryption key is derived deterministically via `scryptSync`.

**Primary recommendation:** Use an in-memory mutex (not a library) for the setup race condition -- the process is single-node and the lock only needs to last milliseconds. Use a GitHub Actions workflow triggered on `push` to `main` that checks whether the README deploy URL matches the current repo and patches it if not.

### Pattern 1: In-Memory Mutex for Setup Race Condition
**What:** A simple boolean flag guarding the setup endpoint so only one request can claim the instance.
**When to use:** Single-process Node.js server where the lock only needs to last for the duration of one HTTP request.
**Why not a file lock:** The current race condition is between concurrent HTTP requests hitting the same Node.js process. Node.js is single-threaded for I/O dispatch, but async operations (like file writes) can interleave. A simple flag set synchronously before any async work prevents this.
**Example:**
```typescript
// In BootService or setup route handler
private setupInProgress = false;

async handleSetup(req, res): Promise<void> {
  // Check 1: Already configured?
  if (this.isConfigured()) {
    res.writeHead(409, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ error: "Instance already claimed" }));
    return;
  }

  // Check 2: Another request in progress?
  if (this.setupInProgress) {
    res.writeHead(409, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ error: "Setup already in progress" }));
    return;
  }

  // Acquire lock synchronously (before any await)
  this.setupInProgress = true;

  try {
    // ... perform setup ...
  } finally {
    this.setupInProgress = false;
  }
}
```

### Pattern 2: Atomic Lock File for Config Write Protection
**What:** An additional filesystem-level guard using `writeFileSync` with `{ flag: 'wx' }` (exclusive create) to prevent concurrent config writes.
**When to use:** Defense-in-depth alongside the in-memory mutex -- ensures that even across process restarts, a partially-written config is detected.
**Example:**
```typescript
import { writeFileSync, unlinkSync } from "node:fs";

const lockPath = join(configDir, ".setup.lock");

// Atomic lock acquisition -- wx flag fails if file already exists
try {
  writeFileSync(lockPath, String(process.pid), { flag: "wx" });
} catch (err) {
  if ((err as NodeJS.ErrnoException).code === "EEXIST") {
    res.writeHead(409);
    res.end(JSON.stringify({ error: "Setup already in progress" }));
    return;
  }
  throw err;
}

try {
  // ... write config ...
} finally {
  unlinkSync(lockPath);
}
```

### Pattern 4: Deterministic Encryption Key from Auth Token
**What:** Derive the encryption key from the auth token using scrypt with a fixed salt.
**When to use:** Replaces the random encryption key stored in auth.json.
**Example:**
```typescript
import { scryptSync } from "node:crypto";

// Fixed salt -- safe because CLAUDEOS_AUTH_TOKEN has high entropy (32-byte secret)
const SCRYPT_SALT = "claudeos-encryption-key-v1";

function deriveEncryptionKey(authToken: string): Buffer {
  return scryptSync(authToken, SCRYPT_SALT, 32);  // 256-bit key
}
```
**Security note:** Using a fixed salt is acceptable here because the input (CLAUDEOS_AUTH_TOKEN) is a 32-byte random secret with ~256 bits of entropy. The salt's purpose in password hashing is to prevent rainbow tables, but that is irrelevant for high-entropy inputs. If desired, a configurable salt could be added later.

### Pattern 5: GitHub Action for Fork README Patching
**What:** A workflow that detects if the deploy button URL points to a different repo than the current one, and patches it.
**Critical finding:** The GitHub Actions `fork` event runs in the **original** repo, not the fork. Workflows are inherited by forks but disabled by default. The correct trigger is `push` to `main` or `workflow_dispatch` -- when a user forks and enables Actions, the first push (or manual trigger) will patch the README.
**Example:**
```yaml
# .github/workflows/patch-fork-readme.yml
name: Patch Deploy Button for Fork
on:
  push:
    branches: [main]
    paths: [README.md]  # Only run when README changes (or on first push)
  workflow_dispatch: {}  # Manual trigger for users who enable Actions

jobs:
  patch:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check and patch deploy button URL
        run: |
          EXPECTED_URL="https://railway.app/new/template?template=https://github.com/${{ github.repository }}"
          CURRENT=$(grep -oP 'https://railway\.app/new/template\?template=https://github\.com/[^)]+' README.md || true)

          if [ -n "$CURRENT" ] && [ "$CURRENT" != "$EXPECTED_URL" ]; then
            sed -i "s|$CURRENT|$EXPECTED_URL|g" README.md
            git config user.name "github-actions[bot]"
            git config user.email "github-actions[bot]@users.noreply.github.com"
            git add README.md
            git commit -m "chore: patch deploy button URL for fork"
            git push
          fi
```
**Important caveat:** GitHub Actions are disabled by default on forks. Users must manually enable them or the workflow never runs. This is a GitHub security feature that cannot be bypassed. The README patching is a convenience -- if users don't enable Actions, they can manually edit the URL or it will still work (Railway will just clone from the wrong repo reference initially, though the forked code is what actually deploys).

### Anti-Patterns to Avoid
- **Reading auth.json for configuration state:** The new model uses env vars. Remove all `auth.json` reads.
- **Storing encryption keys in files:** Derive from `CLAUDEOS_AUTH_TOKEN` via scrypt. No key file storage.
- **Using `proper-lockfile` for HTTP request serialization:** Overkill for a single-process race condition. Use in-memory flag.
- **Assuming fork Actions run automatically:** They do not. Design the system to work without the Action (graceful degradation).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Key derivation | Custom hash function | `crypto.scryptSync()` | Hardened against brute force, built into Node.js |
| AES encryption | Custom cipher logic | `crypto.createCipheriv('aes-256-gcm')` | Already used in codebase, authenticated encryption |
| Lock file creation | Custom open+check | `writeFileSync(path, data, { flag: 'wx' })` | Atomic exclusive-create is an OS primitive |
| YAML/TOML templating | String replacement | Railway template variables `${{secret(32)}}` | Railway handles generation at deploy time |

## Common Pitfalls

### Pitfall 1: TOCTOU in isConfigured() Check
**What goes wrong:** Check `isConfigured()`, then before writing config, another request also passes the check.
**Why it happens:** Node.js event loop can interleave async operations between the check and the write.
**How to avoid:** Set a synchronous boolean flag *before* any async work. The flag must be set in the same microtask as the check.
**Warning signs:** Two successful 200 responses to concurrent setup requests in tests.

### Pitfall 2: scrypt with Weak or Empty Salt
**What goes wrong:** If the salt is empty or predictable AND the input is a weak password, rainbow table attacks are possible.
**Why it happens:** Developer habit from password hashing where random salts are critical.
**How to avoid:** For this use case, the input (`CLAUDEOS_AUTH_TOKEN`) has 256 bits of entropy. A fixed descriptive salt is fine. Document the reasoning.
**Warning signs:** Over-engineering with per-instance random salts that then need persistent storage (re-creating the auth.json problem).

### Pitfall 3: SecretStore Breaking After auth.json Removal
**What goes wrong:** `SecretStore` constructor reads `auth.json` for the encryption key. Removing auth.json breaks all secret operations.
**Why it happens:** SecretStore hard-codes the auth.json path.
**How to avoid:** Modify `SecretStore` (or its initialization) to accept an encryption key directly, derived from `scryptSync(CLAUDEOS_AUTH_TOKEN)`. The `tryCreate` static method should use the env var path.
**Warning signs:** `Error: Auth config not found at /data/config/auth.json` in logs.

### Pitfall 4: GitHub Action Failing Silently on Original Repo
**What goes wrong:** The action runs on the original repo too and tries to "patch" its own already-correct URL, creating unnecessary commits.
**Why it happens:** The `push` trigger fires on every push to main in all repos (original and forks).
**How to avoid:** The comparison check (`$CURRENT != $EXPECTED_URL`) naturally handles this -- if the URL already matches `${{ github.repository }}`, no commit is made.
**Warning signs:** Infinite commit loops from the action pushing, which triggers another push event.

### Pitfall 5: Railway Template Config Conflicts
**What goes wrong:** Both `railway.toml` and `railway.json` exist, causing ambiguity.
**Why it happens:** The project currently has `railway.toml`; adding `railway.json` creates a conflict.
**How to avoid:** Remove `railway.toml` when adding `railway.json`, or stay with `railway.toml`. Do not have both.
**Warning signs:** Railway ignoring your config or using unexpected values.

### Pitfall 6: Cookie Security on HTTP vs HTTPS
**What goes wrong:** Setting `secure: true` on cookies prevents them from working on local HTTP development.
**Why it happens:** Railway uses HTTPS in production, but local dev uses HTTP.
**How to avoid:** Conditionally set `secure` based on environment (e.g., check `NODE_ENV` or presence of Railway-specific env vars).
**Warning signs:** Login works on Railway but cookies aren't set in local dev.

## Code Examples

### Deriving Encryption Key from Auth Token
```typescript
// Source: Node.js crypto docs (https://nodejs.org/api/crypto.html)
import { scryptSync } from "node:crypto";

const SCRYPT_SALT = "claudeos-encryption-key-v1";
const KEY_LENGTH = 32; // 256 bits

export function getEncryptionKey(): Buffer {
  const authToken = process.env.CLAUDEOS_AUTH_TOKEN;
  if (!authToken) {
    throw new Error("CLAUDEOS_AUTH_TOKEN not set");
  }
  return scryptSync(authToken, SCRYPT_SALT, KEY_LENGTH);
}
```

## Open Questions

1. **GitHub Action trigger mechanism for forks**
   - What we know: The `fork` event runs in the original repo, not the fork. Workflows are disabled by default on forks.
   - What's unclear: Whether the CONTEXT.md's "triggers on fork creation event" can be literally implemented. The `push` + comparison approach is functionally equivalent but requires the user to enable Actions.
   - Recommendation: Use `push` to `main` trigger with URL comparison. Document in README that forkers should enable Actions for auto-patching. The deploy button still works without patching -- it just references the original repo URL until patched.

2. **railway.toml vs railway.json migration**
   - What we know: Project currently has `railway.toml`. Railway supports both formats. Template variables use `${{secret()}}` syntax.
   - What's unclear: Whether `railway.toml` supports template variable syntax or only `railway.json` does.
   - Recommendation: Switch to `railway.json` with the `$schema` field for IDE support. Remove `railway.toml`.

3. **SecretStore migration path**
   - What we know: `SecretStore` reads encryption key from `auth.json`. After this phase, auth.json no longer exists.
   - What's unclear: Whether existing encrypted secrets (from v1.0) need migration.
   - Recommendation: Per CONTEXT.md, no migration needed ("no real users on v1.0 yet"). Simply update `SecretStore` to derive key from env var.
