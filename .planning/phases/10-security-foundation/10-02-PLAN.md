  <name>Task 1: Update README deploy button and create GitHub Action for fork patching</name>
  <files>README.md, .github/workflows/patch-fork-readme.yml</files>
  <action>
    **Modify `README.md`:**

    1. Find the deploy button on line 21 (currently `[![Deploy on Railway](https://railway.app/button.svg)](https://railway.app/template/claudeos)`).
    2. Replace with the correct URL for this repo:
       ```markdown
       [![Deploy on Railway](https://railway.app/button.svg)](https://railway.app/new/template?template=https://github.com/aventre-labs/claudeos)
       ```
       Note: Use the actual repo owner/name from the existing README or git remote.

    **Create `.github/workflows/patch-fork-readme.yml`:**

    ```yaml
    name: Patch Deploy Button for Fork
    on:
      push:
        branches: [main]
      workflow_dispatch: {}

    permissions:
      contents: write

    jobs:
      patch:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Check and patch deploy button URL
            run: |
              EXPECTED_URL="https://railway.app/new/template?template=https://github.com/${{ github.repository }}"
              CURRENT=$(grep -oP 'https://railway\.app/new/template\?template=https://github\.com/[^)]+' README.md || true)

              if [ -z "$CURRENT" ]; then
                echo "No deploy button URL found in README.md, skipping"
                exit 0
              fi

              if [ "$CURRENT" = "$EXPECTED_URL" ]; then
                echo "Deploy button URL already correct"
                exit 0
              fi

              echo "Patching deploy button URL..."
              echo "  Current: $CURRENT"
              echo "  Expected: $EXPECTED_URL"

              sed -i "s|$CURRENT|$EXPECTED_URL|g" README.md

              git config user.name "github-actions[bot]"
              git config user.email "github-actions[bot]@users.noreply.github.com"
              git add README.md
              git commit -m "chore: patch deploy button URL for fork"
              git push
    ```

    Key design decisions per CONTEXT.md:
    - Trigger on `push` to `main` (not `fork` event — that runs in the original repo, not the fork).
    - `workflow_dispatch` allows manual trigger for users who enable Actions after forking.
    - The comparison check prevents no-op commits on the original repo (URL already matches).
    - `permissions: contents: write` required for pushing the commit back.

    Notes on Claude's discretion items:
    - `builder: "DOCKERFILE"` — the project has a Dockerfile, Railway should use it (replacing the pre-built image pull from railway.toml's `dockerImage` field).
    - `startCommand` — match the existing entrypoint. Check the Dockerfile for the actual start command and use that. If the Dockerfile has its own CMD/ENTRYPOINT, set `startCommand` to null.
    - `healthcheckPath` and `healthcheckTimeout` — carried over from railway.toml.
    - `variables.CLAUDEOS_AUTH_TOKEN.generator: "secret(32)"` — Railway auto-generates a 32-byte secret at deploy time per `${{secret(32)}}` template variable syntax.
    - Persistent volume config (`/data`) is configured in Railway dashboard, not in railway.json (railway.json does not support volume declarations in the same way railway.toml does).

    **Delete `railway.toml`:**
    Remove the file. Having both creates ambiguity (per research pitfall 5). The railway.json format is needed for template variable support.

    **Important:** Before finalizing railway.json, read the Dockerfile to determine the correct start command. If the Dockerfile already specifies CMD, set `startCommand` to `null` in railway.json.
