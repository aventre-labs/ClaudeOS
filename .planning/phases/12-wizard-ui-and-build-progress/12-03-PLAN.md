    Modify `BootService.serveSetupPage()` to serve the React wizard build output from `wizard-dist/` instead of the single `setup.html` file. The setup server uses raw `http.createServer` on port 8080 (NOT Fastify — see Pitfall 4 from research).

    1. Add a MIME type map at module level:
       ```typescript
       const MIME_TYPES: Record<string, string> = {
         '.html': 'text/html',
         '.js': 'application/javascript',
         '.css': 'text/css',
         '.svg': 'image/svg+xml',
         '.png': 'image/png',
         '.json': 'application/json',
         '.ico': 'image/x-icon',
       };
       ```

    2. Add a helper method `private getWizardDistDir(): string` that resolves the wizard-dist directory path. Try these locations in order:
       - `resolve(this.dataDir, '..', 'wizard-dist')` (container: /app/wizard-dist)
       - `resolve('wizard-dist')` (relative to CWD)
       - `resolve('supervisor', 'wizard-dist')` (development: project root)
       Return the first that exists and contains `index.html`.

    3. Rewrite the `createServer` request handler in `serveSetupPage()`:

       **For `GET /api/v1/*` requests:** Proxy to Fastify on localhost:3100. Use `http.request` to forward the request (method, headers, body) to `http://localhost:3100${req.url}` and pipe the response back. This avoids CORS issues since both servers share the same origin from the user's perspective.

       **For `GET /` or `GET /setup`:** Serve `wizard-dist/index.html`.

       **For `GET /assets/*`:** Serve the matching file from `wizard-dist/assets/`. Determine MIME type from file extension using the MIME_TYPES map. Return 404 if file not found.

       **For `POST /api/v1/*` requests:** Also proxy to Fastify (same proxy logic as GET). The wizard frontend POSTs to /api/v1/wizard/railway/start, etc.

       **For `GET /api/v1/wizard/events` (SSE):** Proxy with special handling — do not buffer the response. Set `Connection: keep-alive` and pipe chunks as they arrive.

       **Keep existing:** The `POST /api/v1/setup` instance claim handler can remain as a direct handler (it doesn't need to proxy since it controls the setup server lifecycle). BUT also proxy it if the wizard frontend calls the Fastify endpoint instead.

       **Fallback:** Any unrecognized GET path serves `index.html` (SPA fallback for client-side routing, though the wizard doesn't use routes).

    4. Remove the old `first-boot/setup.html` file read logic. The fallback HTML ("Setup page not found") can remain as a last resort if wizard-dist doesn't exist.

    5. Ensure the existing health endpoint at `/api/v1/health` is proxied to Fastify (remove the hardcoded response in the setup server, let Fastify handle it).

    6. Run existing boot service tests to verify no regressions.
